# 功能目录

这些能力围绕“网页 → 事实 → 证据 → 内容 → 素材 → 发布包 → 复盘”组织。生成内容是草稿，媒体是草稿或依赖受限的产物；最终流量、转化和平台表现取决于真实账号、平台规则、内容质量和后续运营。

| 功能 | 它做什么 | 解决什么问题 | 给用户带来的收益 | 典型场景 |
| --- | --- | --- | --- | --- |
| 产品网页读取 | 读取公开产品页的正文、标题、结构化信息和浏览器快照，整理成产品事实档案 | 产品信息散落在页面、团队容易漏读或把猜测当事实 | 获得带来源、状态和风险提示的事实底稿，减少返工 | 给一个 SaaS 产品页，先生成定位、功能和目标用户摘要 |
| 多链接与网站产品发现 | 接收多个产品链接，或从网站首页、站点地图、保存的 HTML 中发现候选产品页 | 产品线多时逐页复制链接、人工筛选耗时 | 一次运行建立候选清单，再按评分和边界继续处理 | 运营一个有几十个工具页面的网站，批量准备推广任务 |
| 竞品与爆款内容研究 | 搜索公开或浏览器可见的竞品、创作者和高表现内容，提取钩子、结构、可见指标与来源 | 只凭感觉选题，无法解释参考内容来自哪里 | 得到可复核的证据库、内容模式和后续补采任务 | 上线 AI 工具前，对比 YouTube、知乎、小红书、抖音的公开内容 |
| 本机登录态 Sidecar 研究 | 通过独立的 MediaCrawler Sidecar，在本机 Chrome 登录态下按平台、关键词、详情或创作者模式采集 | 部分平台公开页面不完整，纯匿名读取无法覆盖浏览器可见内容 | 在用户控制的本机环境获得更完整的浏览器可见证据，并保留状态 | 用户在本机完成登录后，补采一个小红书关键词的公开结果 |
| 事实、证据与风险控制 | 为事实、来源、指标、权限和阻塞原因记录状态，区分真实、公开、用户导入、合成演示和缺失 | 草稿中夹杂无法核查的数字、链接或假设 | 审核者知道哪些内容可以用、哪些需要补证据 | 发布前检查引用、风险提示和 `waiting_real_data` 状态 |
| 平台原生文案生成 | 按 YouTube、知乎、小红书、抖音、GitHub 等平台生成标题、正文、标签和互动提示 | 一段通用文案无法适应不同平台的字段和语气 | 减少重复改写，得到可直接审核的多平台草稿 | 同一产品同时准备视频简介、知乎回答和小红书笔记 |
| 视频口播稿与分镜 | 根据产品事实和内容方向生成口播稿、镜头顺序、画面建议、字幕重点和素材清单 | 文案、拍摄和剪辑之间缺少共同结构 | 创作者拿到可拍、可审、可交接的脚本包 | 录制 30–60 秒产品演示，按分镜准备页面录屏 |
| MP4 视频草稿 | 在 FFmpeg 等依赖可用时，把脚本、画面和音频素材渲染为本地 MP4 草稿 | 只有文字无法快速检查节奏和画面缺口 | 在正式拍摄前看到时长、画幅和素材占位结果 | 为短视频首轮审稿生成一个可播放的 MP4 版本 |
| 封面图与详情图 | 在 Pillow 可用时按发布包要求生成 PNG 封面和详情图，并记录尺寸与路径 | 每个平台都要重新整理图片，文件容易漏传 | 图片与标题、视频、发布字段一起交付 | 为小红书或抖音准备一组封面与产品细节图 |
| 完整发布包 | 汇总标题、文案、标签、评论提示、媒体路径、跟踪链接、缺失项、警告和人工步骤 | 内容、素材和发布说明分散在多个文件夹 | 审核者能按一份清单完成准备并留下记录 | 把一项已审核任务交给创作者或客户执行 |
| 受控发布辅助 | 生成 dry-run、官方 API 所需检查、浏览器可见字段填充和发布队列；在最终提交前停止 | 发布步骤容易误操作，凭据和账号权限边界不清 | 先看到完整载荷、截图和缺失项，再由用户决定是否提交 | YouTube 官方 API dry-run，或在小红书创作者页填写可见字段 |
| 发布后真实证据导入 | 从真实 URL、平台导出、截图、评论、订单和收入文件导入证据收件箱 | 发布后数据散落在平台后台，复盘缺少统一入口 | 建立可追溯的发布与业务证据，不把示例数字混入结果 | 发布后放入 `published-urls.csv`、`metrics.csv` 和评论文件 |
| 复盘与下一轮优化 | 读取真实证据，输出表现摘要、受众需求、问题和下一轮选题/钩子建议 | 内容做完后没有结构化反馈，下一轮靠猜 | 用真实数据更新内容方向和优先级 | 比较不同标题的真实指标后规划下一轮素材 |
| Chrome 当前页面转任务 | 用户主动点击后读取当前标签页 URL 和标题，生成可复制的 Skill 或辅助命令 | 从浏览器切到终端时手工抄链接和参数易出错 | 几秒内把正在看的页面变成本地任务，仍可审阅命令 | 在产品官网打开功能页，点击插件后生成单链接运行命令 |
| 中英文界面 | 扩展界面提供中文/英文切换，命令和本地输出仍按所选参数执行 | 团队成员语言不同，使用入口不统一 | 同一套功能可由中文运营和英文协作者共同使用 | 中文运营选择平台，英文协作者审核生成的命令 |
| 本地优先隐私 | 默认把输出、Cookies、Chrome 登录配置和 Sidecar 运行态留在本机 | 公开发行不应携带账号状态、客户数据或隐藏令牌 | 减少数据外传面，便于审计、备份和删除 | 代理团队在客户电脑本地生成材料，再由客户决定分享什么 |

## 统一边界

- Hosted Worker 保持关闭，公开发行不把托管执行描述为可用。
- Cookies 与 Chrome 登录配置只留在本机，不上传到本公开仓库或公开发行包。
- 最终发布需要用户审核并由用户执行确认；浏览器辅助流程在最终提交前停止。
- 工具不会规避 CAPTCHA、平台风险控制、登录检查或账号授权。
- 只接受真实 URL、指标、评论、订单和收入；不会虚构证据。
- 支付、订阅、许可证、点数和账单后端仅排除在功能同步结论之外；扩展既有 billing UI 和 `billing-contract.json` 保留。
